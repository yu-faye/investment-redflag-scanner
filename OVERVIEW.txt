Project: Qben Offensive Audit Red-Flag Scanner

Purpose
- Portfolio-grade personal project that demonstrates offensive audit thinking
  against Middelborg Invest AS core holdings (Qben Infra AB, ININ Group AS).
- Combines structured questions, deterministic detectors, and visualization-ready
  output in one reproducible pipeline.
- Demonstrates ability to move from hypothesis to machine-checkable red flags.

Why this stands out for a candidate
- Frames reporting risk as a testable system, not narrative commentary.
- Demonstrates both accounting intuition and engineering execution.
- Produces explainable outputs that can be reviewed by finance and non-finance stakeholders.
- Targets a real company in Middelborg's portfolio with real public filings.

4-step workflow
1) define questions      -> convert audit hypotheses into structured, reusable checks.
2) ingest numbers/text   -> load report text + curated metric snapshot.
3) run attacks           -> execute deterministic narrative, disclosure, and lag detectors.
4) render red flags      -> emit per-company JSON, follow-up prompts, and dashboard payload.

Detectors implemented (v2)
- lag_causality          : flags claims of action-to-impact with implausibly short lag.
                            v2 adds an IFRS / consolidation guardrail (fully consolidated as of,
                            consolidation of X from <month>, IFRS 3, PPA, business combination, etc.)
                            so mechanical accounting inclusion is tagged as "info" rather than as a
                            synergy promise. See src/detectors/lag_causality.py.
- narrative_dissonance   : flags optimistic narrative families that contradict directional metrics.
                            v2: per-metric expected_direction + neutral_band (leverage = high bad).
- selective_disclosure   : flags KPI drops or new reclassification frames vs prior period.
                            v2: prior periods wired in for Qben, ININ, Techstep, K33, North Energy.
- stress_test_prompts    : attaches English follow-up questions to every red flag
                            (bilingual EN+ZH schema retired in v7).

Engine invariants (the structural protections):
  1. Single-source absence cannot drive critical. A hypothesis with
     evidence from <=1 distinct tap caps at warning. This is the
     structural fix to the "what if the database just didn't collect it"
     critique that v8 was vulnerable to.
  2. Peer-control rule. Refuting evidence is only treated at full weight
     when at least one peer hypothesis confirmed via the same tap_id
     (proving the tap reaches this entity class). Without a passing peer,
     refuting evidence is downgraded one severity band.
  3. Falsification coverage. Critical additionally requires that every
     falsification_question marked blocking_for_critical=true has at
     least one EvidenceEntry addressing it.
  4. Engine is the only writer of derived_severity. Taps emit verdicts
     only (confirms / partial / refutes / not_found / neutral / error);
     never severity.

v11 sprint kickoff (planning only -- not implemented in this iteration)
====================================================================
Goal of v11 is to demonstrate that the v9/v10 architecture scales from
"one issuer, four subsidiary claims, three primary taps + two derived
taps" to "a portfolio of 100 Nordic issuers, multi-country tap coverage,
5 AI-suggested pattern hypotheses, cross-issuer clustering." Nothing in
v11 changes the engine contract or the dashboard contract -- only the
breadth of hypotheses, the breadth of taps, and the new cross-issuer
analytics layer.

Below is the kickoff scope so a future sprint can pick this up cleanly.

C.1  100-Nordic-company universe (data source)
- Primary: OMX Nordic Small Cap + First North Stockholm + Oslo Bors
  Euronext Growth + Helsinki First North = ~700 listings. Filter to the
  ~100 with public-procurement-exposed segments (NACE/SNI codes 41-43
  construction, 35 power, 38-39 environmental, 49-52 transport, 86
  healthcare, 80 security).
- Filter to issuers that publish English or Nordic-language interim
  reports as PDF on issuer-controlled URLs (not paywalled databases).
- Output: validation/universe_v11.json with company_id, exchange,
  ticker, NACE_segment, primary_geography, ir_pdf_landing_url,
  ir_archive_root. Same schema discipline as data/sources.json so the
  existing ingest pipeline picks it up unchanged.
- This file is a JSON edit, not a code change -- the same property as
  validation/hypotheses.json in v9/v10.

C.2  Multi-country tap matrix (extending the v9/v10 EvidenceTap pattern)
- New primary taps, each ~80 lines like ted_tap.py, no engine changes:
  * uk_procurement     : Contracts Finder + Find a Tender (UK)
  * de_procurement     : evergabe-online + bund.de TED-DE feed
  * dk_procurement     : Udbud.dk (Denmark)
  * fi_procurement     : Hilma (Finland)
  * se_procurement     : tendsign / Mercell Sweden public feed
  * uk_company_registry: Companies House API (free; analogous to BRREG)
  * de_company_registry: Unternehmensregister.de (limited free tier)
  * insider_trading    : Finanstilsynet NewsWeb (NO); Finansinspektionen
                         insynshandel (SE); BaFin DEA notifications (DE)
  * media              : MFN (SE), CISION wires (NO/SE/FI), GlobeNewswire
                         Nordic, EQS Group filings
- New derived taps (cross-issuer, second pass like v10 derived_*):
  * cross_issuer_supplier_overlap : detects when issuer A and issuer B
    both claim the same subcontractor as a "specialist"; if Doffin shows
    that supplier mostly working for B, A's claim is weakened.
  * insider_vs_kpi_window         : compares insider sell volume in a
    +/-30d window around the report publication date against the
    direction of the KPI move. Net selling into a KPI deterioration ->
    refutes "long-term confidence" narrative.
  * peer_segment_calibration      : for issuers in the same NACE segment
    and overlapping geography, computes the segment-level external-
    support-pct distribution; any single issuer falling >2sigma below
    the peer median refutes its own revenue_pipeline_support claim
    independently of its absolute threshold.

C.3  Five AI-pattern hypotheses (the "AI Product Engineer" demo deliverable)
The system already has 1 AI-shaped hypothesis category in v10
(revenue_pipeline_support). v11 adds 5 more, each registered in
validation/hypotheses.json with the same schema discipline (claim,
claim_metadata, falsification_questions, peer_controls) so the engine
treats them identically. Each of these is a *pattern* an LLM was used
to surface offline (one-shot, not in the live pipeline) and then
hard-coded back into the deterministic system.

  P1  segment_reframe_under_pressure
      An issuer that previously reported segment X drops it from the
      KPI table after segment X deteriorates. Falsifiable by: presence
      of segment X dimension in prior periods, absence in current,
      simultaneous narrative reference to "simplified reporting" or
      "focus on core segments". Derived tap reads sources.json
      historical entries and compares per-segment line presence.

  P2  one_off_dressed_as_recurring
      A headline profit driven by a one-off gain (divestment, fair-
      value remeasurement, settlement) is presented in the headline
      without parenthetical adjustment, while the discontinued-ops
      breakdown is buried. v10 already saw this on Qben Q1 2026
      (147 MSEK headline driven by Rail divestment; continuing-ops
      result -53 MSEK). Falsifiable by: ratio of headline-profit to
      continuing-ops-profit, presence of the word "divestment" /
      "avyttring" / "frasalg" in the same paragraph as the headline
      number, absence of an adjusted-EBITA breakdown.

  P3  acquisition_lag_revenue_attribution
      A revenue increase is attributed to "organic growth" while the
      time window includes an acquisition consolidation date.
      Falsifiable by: M&A press release date < period end, presence
      of "consolidated as of" / "from <month>" / "PPA" in the same
      paragraph, magnitude of acquired-entity revenue >= delta vs
      prior period.

  P4  capex_amortisation_mismatch
      Capex << depreciation for >=4 consecutive periods, while the
      narrative emphasises "continued investment" / "growth capex".
      Falsifiable by: capex_to_depreciation_ratio time series, presence
      of growth-capex tokens in CEO comments, peer-segment capex ratio.

  P5  insider_sells_into_optimism
      Net insider selling in the +/-30d window around publication
      coincides with above-median positive-framing density in the
      CEO comment. Cross-tap derived: needs insider_trading tap
      (NewsWeb/insynshandel) + ExplanatorySlippageScanner reused with
      an aggregate-density input.

Each of P1-P5 is one new hypothesis category + one new derived tap.
No engine change is required beyond a single _apply_<category>_rule
helper if the category needs an R7-style override (e.g. P5 might).

C.4  Cross-issuer clustering layer
- New module src/analytics/cross_issuer.py (planned, not implemented):
  loads outputs/argument_trees.json for every issuer in universe_v11,
  extracts (issuer, claim_category, refuted_falsification_question)
  triples, builds a similarity graph (issuers as nodes, shared
  refuted question as edges weighted by Jaccard), and emits
  outputs/issuer_clusters.json.
- New dashboard panel (planned): "Pattern landscape" -- a UMAP /
  force-directed layout of issuers coloured by dominant refuted
  pattern. Click an issuer -> drills into that issuer's leaderboard;
  click a cluster -> Sankey filters to that cluster's findings.
- New Streamlit panel (planned): peer-percentile cards showing
  external_support_pct, positive_framing_density, capex_amort_ratio,
  net_insider_flow versus the issuer's NACE-segment peer distribution.

C.5  v11 success criteria (acceptance, not implemented now)
- pipeline runs against >= 100 issuers without per-issuer code changes
- >= 5 of the 6 hypothesis categories (revenue_pipeline_support + P1-P5)
  produce at least one critical finding across the universe
- the global Sankey at the top of the dashboard shows >= 4 distinct
  derived_analysis lanes
- cross_issuer_supplier_overlap surfaces >= 1 pair of issuers sharing
  a contested subcontractor
- the docs deck (RESULTS.md + OVERVIEW.txt) gains a v11 section
  with a worked example for one P1-P5 pattern (analogous to the
  v10 worked example for revenue_pipeline_support)

C.6  v11 explicit non-goals
- NO LLM in the live pipeline. LLMs may be used offline to surface
  pattern candidates (P1-P5) but each pattern is then hard-coded into
  a deterministic detector before it goes into the run.
- NO change to the engine verdict vocabulary or the severity bands.
- NO change to the dashboard contract; the new "Pattern landscape"
  panel is additive.
- NO web scraping where a documented public API exists. Every new
  tap must follow the v9/v10 EvidenceTap contract (cache raw response
  with sha256 + fetched_at_utc + query_url; emit a single
  EvidenceEntry verdict).
